# YouTube Script Framework Library

Two structural skeletons extracted from high-performing YouTube videos. When writing a new script, select the framework that matches the video format and map the content into it beat-by-beat. Do not deviate from the structure.

---

## Framework Selection

| Video type | Use |
|---|---|
| Tool tutorial, system walkthrough, "lazy/easy/fast way to do X", software demo, process SOP | Framework 01 |
| Business audit, case study teardown, live coaching, consulting explainer, "what I'd do differently" | Framework 02 |

---

## Framework 01: Elimination Funnel Reveal

**Label:** `ELIMINATION FUNNEL → WITHHELD REVEAL → LIVE PROCESS WALKTHROUGH`

---

### 1. Hook Pattern

**Type:** Honesty disarm → Bold promise → Credibility anchor

Open by killing the obvious objection first ("there's no magic solution"). Immediately pivot with a contrarian reframe ("but there IS a lazy/faster/easier way"). Anchor with a specific result claim — revenue number, client count, timeframe. The honesty in sentence one makes the promise in sentence two land harder.

**Trigger:** Pattern interrupt + curiosity gap.

---

### 2. Transition to Core Problem

**Type:** Exhaustion of alternatives → Elimination funnel

Walk through every known competitor method (minimum 5). Give each one a "disqualified" verdict with a single concrete reason — labor, cost, skill gap, time, volume dependency. Build cumulative frustration. End by naming the one method not yet covered.

**Tension mechanic:** False choice collapse — all roads are blocked except one.

---

### 3. Content Delivery Structure

**Type:** Problem stack → Solution reveal → Sequential how-to

| Beat | What happens |
|---|---|
| Beat 1 | Alternatives debunked — disqualification ladder, 5–7 methods eliminated |
| Beat 2 | Solution named and framed around the core promised value |
| Beat 3 | Why this solution fits the promised frame (lazy/fast/easy) — specific attributes |
| Beat 4 | Sequential how-to — numbered steps, each anchored to one tool or one action |
| Beat 5 | Live demo or walkthrough per step — proof of ease in real time |
| Beat 6 | Scalability bridge — how to delegate, hire for, or grow beyond the system |

---

### 4. Retention Mechanics

- **Hook-level open loop:** Withhold the solution name through the entire elimination sequence
- **Step-level micro loops:** Each step ends with a bridge to the next — never resolve and stop, resolve and bridge
- **Pattern interrupt:** Self-questioning device every 3–4 minutes ("But [name], how do I do this?") — breaks lecture mode, resets attention
- **Curiosity gap:** Frame each feature or step as something most people don't know exists before revealing it

---

### 5. Proof / Credibility Layer

- Early authority via personal result metric — specific numbers, not vague claims
- Trust reinforced through failure disclosure — methods tried and abandoned, with honest reasons why
- Live demo as implicit proof — viewer watches it work, not just hears about it
- No external social proof needed — first-person experience plus demonstration is sufficient

---

### 6. CTA Architecture

**Type:** Skepticism acknowledgment → Continuation CTA

Name the remaining objection explicitly. Offer the next video as its resolution. Frame the CTA as the logical next step for the doubtful, not a push. Trigger line: not "if you enjoyed this" — use "if you're still not sure."

---

### 7. Pacing & Tone

- Short declarative sentences. No paragraph monologues.
- Heavy first-person past tense — "I tried this, here's what happened"
- Self-interruption device used as pacing reset every 3–4 minutes
- Casual, slightly self-deprecating — the method's simplicity is a badge, not an apology
- Demo sections shift to slower, narrated walkthrough tone
- Rhythm: Statement → Consequence → Bridge. Repeat.

## Framework 02: Live Diagnosis Rebuild

**Label:** `LIVE DIAGNOSIS → ROOT CAUSE FLIP → ASSET REBUILD → NUMBERED EXECUTION RECAP`

---

### 1. Hook Pattern

**Type:** Third-party problem introduction + credibility anchor + stakes declaration

Open on the subject, not the host. Introduce the subject with a specific revenue number and a named problem. Establish host authority via a direct personal parallel to that exact situation. Declare what changes if this gets fixed.

**Trigger:** Proxy identification — viewer sees themselves in the subject before they see the expert.

---

### 2. Transition to Core Problem

**Type:** Surface symptoms → Root cause excavation

Start with the subject's stated problems — usually 2–3, disconnected. Host interrupts the surface diagnosis and names the single structural constraint underneath all of them. The gap between what the subject believes and what's actually true pulls viewers forward.

**Tension mechanic:** Subject thinks they have 3 problems. Host reveals they have 1.

---

### 3. Content Delivery Structure

**Type:** Diagnosis → Asset inventory → Rebuild → Execution plan

| Beat | What happens |
|---|---|
| Beat 1 | Current state mapped — revenue, team size, offers, acquisition method, all quantified |
| Beat 2 | Problem mis-diagnosis exposed — stated problems vs. actual root cause |
| Beat 3 | Asset inventory — list everything available before rebuilding anything |
| Beat 4 | Offer reconstruction — strip, reframe, repackage using existing assets only |
| Beat 5 | Acquisition fix — front-end rebuilt to match the new offer |
| Beat 6 | Scaling path — reverse-engineer the revenue ceiling and show how to remove it |
| Beat 7 | Numbered recap — every change restated with its expected multiplier |

---

### 4. Retention Mechanics

- **Tension loops:** Each section ends with an unresolved constraint before moving to the next fix
- **Pattern interrupt:** Host stops subject mid-answer to name the real issue — often a single blunt word
- **Meta-commentary:** Host narrates his own thinking process out loud mid-consultation — turns advice into a teaching layer running simultaneously
- **Framework drops:** Named models inserted as conceptual anchors mid-conversation — each one resets attention
- **Analogy injections:** Concrete analogies used to collapse complex ideas before moving forward

---

### 5. Proof / Credibility Layer

- Authority established via aggregate portfolio metric — companies, revenue, scale
- Credibility reinforced through specific operational details — exact ad spend, exact numbers, not general expertise claims
- Live problem-solving is the primary proof vehicle — competence demonstrated in real time
- Subject's own data used as evidence against their current beliefs — the most powerful form of proof because they cannot deny their own numbers

---

### 6. CTA Architecture

**Type:** Resource embed mid-video + next video continuation CTA

Two CTAs, neither isolated at the end. Mid-video: free resource offered at a natural diagnostic inflection point, framed as a gift inside the content flow. End: next video teased as a different subject applying the same method — expanded learning, not subscription bait.

---

### 7. Pacing & Tone

- Dialogue-driven — subject's responses control pacing, host corrects and redirects
- Short declarative verdicts after long diagnostic exchanges — rhythm contrast signals importance
- Host narrates framework application out loud — dual-track content throughout
- Direct, not dismissive — disagreement delivered as obvious fact, not personal opinion
- Rhythm: Long diagnostic exchange → Single blunt verdict → Bridge to next beat. Repeat.
